## Цель

Настроить Ouroboros для выполнения конкретных корпоративных задач, **сохранив** его уникальные свойства: самоэволюцию, непрерывную идентичность, фоновое мышление, мульти-модельный контроль качества, версионность через Git.

## 4. Добавление корпоративных инструментов (tools)

### 4.1. Архитектура плагинов Ouroboros

Ouroboros использует **авто-обнаружение инструментов**: каждый Python-модуль в ouroboros/tools/, который экспортирует функцию get_tools() → List[ToolEntry], автоматически регистрируется при запуске.

**Из registry.py (строки 217-241):**
- В development-режиме: pkgutil.iter_modules сканирует все модули в ouroboros/tools/ (кроме __init__ и registry)
- В frozen/packaged-режиме: используется фиксированный список _FROZEN_TOOL_MODULES (строка 211)

### 4.2. Пошаговая инструкция: создание нового инструмента

**Шаг 1.** Создать файл в ouroboros/tools/, например **ouroboros/tools/crm_integration.py**

**Шаг 2.** Реализовать структуру ToolEntry:

| Поле ToolEntry | Тип | Описание |
|---------------|-----|----------|
| **name** | str | Уникальное имя инструмента (snake_case) |
| **schema** | dict | JSON Schema: name, description, parameters (OpenAI function calling format) |
| **handler** | Callable | Функция fn(ctx: ToolContext, **kwargs) → str |
| **is_code_tool** | bool | True если инструмент выполняет код (по умолчанию False) |
| **timeout_sec** | int | Таймаут выполнения (по умолчанию 360 сек) |

**Шаг 3.** Экспортировать get_tools():

| Элемент | Пример |
|---------|--------|
| Импорт | from ouroboros.tools.registry import ToolContext, ToolEntry |
| Handler | def _query_crm(ctx: ToolContext, customer_id: str, fields: str = "") → str: ... |
| get_tools() | return [ToolEntry("query_crm", {schema}, _query_crm, timeout_sec=30)] |

**Шаг 4.** Если инструмент должен быть доступен с раунда 1 — добавить его имя в **CORE_TOOL_NAMES** в ouroboros/tool_capabilities.py.

**Шаг 5.** Если инструмент должен быть доступен в фоновом режиме — добавить его имя в **_BG_TOOL_WHITELIST** в ouroboros/consciousness.py (строка 490).

**Шаг 6.** Для frozen/packaged сборки — добавить имя модуля (без .py) в **_FROZEN_TOOL_MODULES** в ouroboros/tools/registry.py (строка 211).

**Шаг 7.** Обновить docs/ARCHITECTURE.md и docs/CHECKLISTS.md (если нужна проверка при ревью).

### 4.3. Контекст выполнения (ToolContext)

Каждый handler получает ToolContext (из registry.py) с полезными атрибутами:

| Атрибут | Тип | Описание |
|---------|-----|----------|
| **ctx.repo_dir** | pathlib.Path | Путь к Git-репозиторию |
| **ctx.drive_root** | pathlib.Path | Путь к ~/Ouroboros/data/ |
| **ctx.emit_progress_fn** | Callable | Отправка прогресса пользователю |
| **ctx.pending_events** | list | Очередь событий (для трекинга бюджета) |
| **ctx.task_id** | str | ID текущей задачи |
| **ctx.current_task_type** | str | Тип задачи (task/evolution/review/consciousness) |
| **ctx.repo_path(rel)** | метод | Безопасное разрешение пути внутри repo (с проверкой boundaries) |
| **ctx.drive_path(rel)** | метод | Безопасное разрешение пути внутри data (с проверкой boundaries) |

### 4.4. Безопасность кастомных инструментов

| Аспект | Как обеспечить |
|--------|---------------|
| **Проверка безопасности** | Инструменты с именами из CHECKED_TOOLS (safety.py) проходят LLM-проверку. Новые инструменты НЕ проверяются автоматически — добавьте имя в CHECKED_TOOLS если нужна проверка |
| **Path traversal** | Используйте ctx.repo_path() и ctx.drive_path() — они проверяют, что путь не выходит за границы |
| **Секреты** | Не выводите API keys, токены, пароли в результатах инструментов |
| **Timeout** | Устанавливайте адекватный timeout_sec для внешних вызовов |

### 4.5. Пример: инструмент интеграции с CRM

| Элемент | Содержимое |
|---------|-----------|
| **Файл** | ouroboros/tools/crm_integration.py |
| **Инструменты** | query_crm (чтение), update_crm_record (запись), search_crm (поиск) |
| **Зависимости** | pip install requests (или SDK вашей CRM) |
| **Безопасность** | query_crm → не требует проверки (read-only). update_crm_record → добавить в CHECKED_TOOLS в safety.py |
| **Доступность** | Добавить в CORE_TOOL_NAMES если нужен с раунда 1 |

## 5. Инструменты, требующие доступа в интернет

**Это критическое ограничение для корпоративного контура.** Ниже — полный перечень компонентов Ouroboros, которым нужен интернет:

### 5.1. Вызовы LLM (ouroboros/llm.py) — ОСНОВНОЕ

| Провайдер | URL | Ключ | Когда используется |
|-----------|-----|------|-------------------|
| **OpenRouter** | https://openrouter.ai/api/v1 | OPENROUTER_API_KEY | Основной маршрут всех LLM-вызовов (задачи, safety checks, consciousness, review) |
| **Anthropic** (direct) | https://api.anthropic.com/v1 | ANTHROPIC_API_KEY | При настройке прямого доступа + Claude Code CLI |
| **OpenAI** (direct) | https://api.openai.com/v1 | OPENAI_API_KEY | При настройке прямого доступа |
| **Cloud.ru** | https://foundation-models.api.cloud.ru/v1 | CLOUDRU_FOUNDATION_MODELS_API_KEY | Российский провайдер |
| **OpenAI-compatible** | Кастомный URL | OPENAI_COMPATIBLE_API_KEY | Любой совместимый эндпоинт |
| **Локальная модель** | http://localhost:{LOCAL_MODEL_PORT} | — | Через llama-cpp-python, **БЕЗ интернета** |

**Для корпоративного контура без интернета:** Использовать локальную модель (USE_LOCAL_MAIN=True и т.д.) или внутренний прокси (LiteLLM, gpt2giga) с маршрутом на on-premise модели.

| **compact_context, codebase_health, codebase_digest** | compact_context.py, health.py, core.py | Локальный анализ |

**Важно:** Инструменты analyze_screenshot, advisory_pre_review, review_status, multi_model_review — технически локальные, но вызывают LLM API через llm.py, поэтому требуют доступ к LLM-провайдеру.

### 5.4. Рекомендации для закрытого корпоративного контура

| Задача | Решение |
|--------|---------|
| **LLM без интернета** | Локальная модель через llama-cpp-python (Metal на Apple Silicon, CPU на Linux) или on-premise vLLM с открытыми моделями |
| **LLM через прокси** | LiteLLM как внутренний прокси → маршрутизация на корпоративный API. Настроить OPENAI_COMPATIBLE_BASE_URL |
| **Web search** | Создать кастомный инструмент corporate_search в ouroboros/tools/ с вызовом внутреннего Elasticsearch/Meilisearch |
| **GitHub** | Заменить github.py на интеграцию с внутренним GitLab/Gitea. Настроить GITHUB_REPO на внутренний сервер |
| **Browser** | Ограничить разрешённые домены в browse_page handler или отключить |

## 8. Что ОБЯЗАТЕЛЬНО сохранить (ядро самоэволюции)

Следующие механизмы — уникальное преимущество Ouroboros. Их удаление превратит систему в обычного чат-бота:

| # | Механизм | Файл(ы) |
|---|----------|---------|
| 1 | **7-шаговый протокол эволюции** (assess → select → implement → smoke test → multi-model review → Bible check → commit+restart) | SYSTEM.md |
| 2 | **Background consciousness loop** (проактивное самосовершенствование между задачами) | consciousness.py + CONSCIOUSNESS.md |
| 3 | **Identity persistence** (накопление опыта и самопознания) | identity.md + update_identity |
| 4 | **Pattern register** (системная работа над ошибками) | knowledge/patterns.md |
| 5 | **Knowledge base** (растущая база доменных знаний) | knowledge/ + knowledge tools |
| 6 | **Multi-model review** (перекрёстная проверка изменений) | review.py + CHECKLISTS.md |
| 7 | **Task reflections** (рефлексия после ошибок) | reflection.py + task_reflections.jsonl |
| 8 | **Scratchpad** (рабочая память между задачами) | scratchpad_blocks.json |
| 9 | **Git-версионность** (полная прослеживаемость) | git.py + repo_commit |
| 10 | **4 вопроса самодиагностики + Drift Detector** (предотвращение деградации) | SYSTEM.md |

**Принцип адаптации: менять ЦЕЛИ и КОНТЕКСТ, сохраняя МЕХАНИЗМЫ.** Ouroboros ценен не тем, что он делает сейчас, а тем, КАК он учится делать лучше.
